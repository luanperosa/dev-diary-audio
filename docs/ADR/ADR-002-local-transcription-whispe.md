ADR-002 — Local Transcription with Whisper (Daily Batch)

Status: Implemented
Date: 2026-01-24
Implementation Date: 2026-01-24
Phase: Phase 1.5 — Local Transcription & Raw Transcript Output

Context

The application already supports:

Long audio recording sessions

Multiple audio files per day

Local audio storage organized by date, e.g.:

~/.config/dev-diary-audio/recordings/
  └── 2026-01-24/
      ├── recording-09-12.webm
      ├── recording-11-03.webm
      └── recording-15-40.webm


The next step is to enable local transcription of all recordings from a given day, producing a single raw transcript that can later be used for diary generation.

Constraints:

Transcription must work for long sessions

Mixed language (Portuguese + English + any language) must be handled automatically (Whisper can deal with it)

No external API dependency

No real-time transcription required

Decision
Transcription Strategy

Use Whisper CLI (open-source) as an external process.

Whisper runs outside Electron, invoked via system command.

Electron acts as an orchestrator, not an ML runtime.

Rationale:

Whisper is mature and reliable

CLI usage avoids embedding ML logic in Electron

Simplifies cross-platform support

Enables offline usage

Avoids API limits and recurring costs

Transcription Scope

Transcription is triggered manually by the user via a per-day button embedded in each day's recordings section.

UI Design:
- Each day's recordings section includes inline transcription controls
- If transcript exists: Shows transcript in textarea (expanded by default) with "Hide Transcript" toggle
- If no transcript: Shows "Generate Transcription for [date]" button

When triggered, the app:

Identifies the selected date

Collects all audio files from that date's folder

Runs Whisper on each audio file

Produces raw transcription output (JSON)

Aggregates results into a single daily transcript file

Automatically reloads and displays the new transcript

This is a batch process, not real-time.

UI/UX Design Decision

Transcription controls are embedded within each day's recordings section (not a separate top-level UI).

Rationale:
- Better contextual organization - transcripts appear next to their source recordings
- One-click transcription per day - no need to select dates from a picker
- Clear visual association between recordings and transcripts
- Automatic display of existing transcripts on app load
- Simpler user workflow

Data Persistence & Loading

Transcripts are stored in the date folder: `~/.config/dev-diary-audio/recordings/YYYY-MM-DD/transcript-YYYY-MM-DD.txt`

On app startup, the system automatically:
- Scans each date folder for any `.txt` file
- Loads transcript content if found
- Displays it in the UI (expanded by default)

Transcript files persist across app restarts

Data Structure (Backend):
```javascript
recordings[date] = {
  sessions: [{ sessionName, filename, path, size, modified }, ...],
  transcript: { path, content, filename } | null
}
```

This clear separation makes transcript detection and display straightforward

Audio Processing Model

Each audio file is transcribed independently

Whisper’s internal chunking is relied upon

No custom audio chunking logic is introduced at this phase

Rationale:

Simpler implementation

Easier failure recovery

Aligns with Whisper’s native behavior

Whisper Invocation

Whisper is executed using a CLI command similar to:

whisper <audio-file> \
  --model small \
  --output_format json \
  --output_dir <recordings-date-folder>


Language detection is automatic

Model choice is configurable later

Output Format

JSON is the source of truth

One JSON file per audio file

A merged daily raw transcript is generated by concatenating segments in chronological order

The merged transcript is stored locally and used as the primary artifact for later processing.

Frontend Behavior

UI provides a button:

“Generate transcription of this day”

While transcription is running:

Show loading / progress indicator

Disable duplicate actions

After completion:

Display the full raw transcript

Transcript is initially collapsed

User can expand/collapse the transcript view

No editing, formatting, or summarization occurs at this stage.

Process Responsibilities
Renderer Process

User interaction

Trigger transcription request

Display transcription status

Render raw transcript output

Main Process

Resolve file paths

Spawn Whisper processes

Track execution status

Read transcription output files

Assemble daily transcript

Whisper (External)

Perform speech-to-text

Detect language

Generate segment-level transcription

Consequences
Positive

Fully offline transcription

No duration limits

Handles mixed languages naturally

No recurring cost

Clear separation of concerns

Reusable transcription artifacts

Trade-offs

Transcription is not instant

CPU-intensive process

Requires local Whisper installation

This ADR intentionally favors:

Simplicity

Determinism

Local-first architecture

## Implementation Summary

**Status: Fully Implemented (2026-01-24)**

### Completed Features

**Whisper Installation Check**
- Detects if Whisper CLI is available on system startup
- Shows warning banner with installation instructions if missing
- Works around Whisper's lack of `--version` flag by checking `--help` output

**Per-Day Transcription UI**
- Embedded within each day's recordings section
- Context-aware: transcription controls appear next to source recordings
- Auto-detects existing transcripts on app load
- No date picker needed - one button per day

**Data Persistence**
- Transcripts stored as `transcript-YYYY-MM-DD.txt` in date folder
- Auto-loaded on app startup by scanning for `.txt` files in date folders
- Data structure: `recordings[date] = { sessions: [...], transcript: {...} }`

**Transcription Workflow**
1. User clicks "Generate Transcription for [date]" button
2. App scans date folder for all `.webm` files
3. Whisper processes each file: `whisper <file> --model small --output_format json --output_dir <date-folder>`
4. App merges JSON segments into single text file with timestamps
5. UI automatically reloads and displays new transcript

**UI States**
- **No transcript**: Shows "Generate" button with status area
- **Transcript exists**: Shows "✓ Transcript available", displays content in textarea, provides "Hide" toggle
- **Processing**: Shows progress per file, disables buttons
- **Complete**: Auto-reloads list, shows transcript expanded

**Error Handling**
- Whisper not installed (startup warning)
- No recordings found for date
- Individual transcription failures (partial success supported)
- IPC communication errors
- File system errors

### Technical Decisions

**Content Security Policy Compliance**
- Uses event listeners with `data-` attributes instead of inline `onclick` handlers
- Buttons have CSS classes (`generate-transcript-btn`, `toggle-transcript-btn`)
- Events attached after HTML rendering via `attachTranscriptionEventListeners()`

**Transcript Format**
```
# Developer Diary Transcript - 2026-01-24

## session-09-30-15

[00:15] First segment text...
[01:23] Second segment text...

## session-14-20-30

[00:05] Another segment...
```

**File Locations**
- Audio: `~/.config/dev-diary-audio/recordings/2026-01-24/session-XX-XX-XX/recording.webm`
- JSON: `~/.config/dev-diary-audio/recordings/2026-01-24/session-XX-XX-XX/recording.json`
- Transcript: `~/.config/dev-diary-audio/recordings/2026-01-24/transcript-2026-01-24.txt`

### Known Limitations & Future Work

- No real-time transcription progress updates (processes run sequentially)
- UI needs refinement for better user feedback during long transcriptions
- No cancellation support once transcription starts
- Whisper model is hardcoded to 'small' (could be configurable)
- No retry mechanism for failed individual file transcriptions
- README documentation for Whisper installation deferred
